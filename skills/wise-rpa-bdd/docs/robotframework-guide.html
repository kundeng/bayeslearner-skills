0 The Problem WiseRpaBDD Solves

Web scraping is a solved problem — for one site, one time

Any developer can write a Python script that scrapes a website. The problem is what happens after:

Pain pointWhat goes wrong
VarianceEvery site requires custom code. A Yelp scraper shares almost nothing with a Splunk docs scraper. Teams accumulate dozens of bespoke scripts, each with its own patterns, error handling, and selector conventions.
FragilityA site redesign breaks the script. Nobody remembers which selectors were chosen or why. The person who wrote it has moved on. Fixing it means reverse-engineering the original exploration all over again.
OpacityA scraping script is a black box to anyone who didn't write it. A business user cannot open scrape_yelp.py and understand what it collects, how it paginates, or what happens if a field is missing. They depend entirely on the developer.
RepeatabilityRunning the scraper again on a different date, with different auth, or against a staging version requires code changes. There's no clean separation between what to scrape and how to scrape it.
Communication gapThe user says "get me all the laptop prices." The developer writes 200 lines of Playwright. The user gets a JSON file and asks "where are the HDD variants?" There's no shared artifact that both parties can review before the scrape runs.

The core insight: separate the spec from the engine

WiseRpaBDD resolves this by splitting scraping into two distinct layers:

LayerWho owns itWhat it looks like
Spec (the .robot file)The user, the AI agent, or a QA analystHuman-readable BDD steps: "When I paginate by next button up to 50 pages", "Then I extract fields ..."
Engine (WiseRpaBDD.py)The framework maintainerA generic execution engine that interprets the spec using browser automation

The spec is site-specific but code-free. The engine is generic but powerful. A new site means a new .robot file, not a new Python script.

What this gives you

  • No variance — every scrape uses the same keywords, the same structure, the same engine. A quotes scraper and a Splunk docs scraper differ only in their selectors and rule tree, not in their code patterns.
  • Readable specs — a business user can open a .robot file and understand what it does. "When I expand over elements article.product_pod" is self-explanatory. The *** Comments *** section documents the exploration evidence.
  • Repeatable execution — the same .robot file produces the same results on any machine with the engine installed. Variables parameterize URLs, credentials, and page limits without code changes.
  • Built-in quality assurance — quality gates, BDD validation, and dryrun checks catch problems before the scrape runs. The user knows upfront whether the spec is well-formed.
  • Shared language — the .robot file is the contract between the user who requests data and the agent (AI or human) who builds the scraper. Both can read it, review it, and iterate on it.
Key Concept
Traditional scraping: "write custom code per site." WiseRpaBDD: "write a declarative spec per site, execute it with one generic engine." The spec is the product — a portable, readable, tested extraction plan that a non-developer can run, inspect, and maintain.

The AI generation angle

Because the spec is structured and the keyword catalog is finite (~60 keywords), an AI agent can generate the spec from a natural-language requirement. The agent explores the live site, gathers selector evidence, and produces a tested .robot file. The user reviews and refines the spec in plain English, not in code. This is the explore → draft → review → ship workflow at the heart of the wise-rpa-bdd skill.

1 What is Robot Framework

The short version

Robot Framework (RF) is an open-source keyword-driven test automation framework written in Python. Instead of writing test logic in raw Python (as you would with pytest or unittest), you write tests in a human-readable tabular syntax inside .robot files.

How it differs from pytest

Aspectpytest / unittestRobot Framework
LanguagePure PythonTabular .robot files (calls Python under the hood)
AbstractionFunctions / classesKeywords — named actions that read like English sentences
AudienceDevelopersAnyone who can read structured English (QA, analysts, agents)
ReportsTerminal + pluginsBuilt-in HTML reports (log.html, report.html)
Librariespip packagesKeyword libraries (Python classes with @keyword decorators)

Why the wise-rpa-bdd project uses RF

The project needs a declarative scraping language that an AI agent can both read and generate. RF's keyword syntax is close to natural language, so a Claude agent can produce a valid .robot file without understanding Python internals. The actual browser automation still happens in Python — RF is the glue layer.

Key Concept
RF is not a replacement for Python. It is a coordination layer that calls into Python keyword libraries. Think of it as a domain-specific language (DSL) for test/automation plans.

2 .robot File Anatomy

A .robot file is divided into named sections, each starting with three asterisks: *** Section Name ***.

The four main sections

*** Settings ***
# Configuration: which libraries to load, setup/teardown hooks
Library           Browser
Library           WiseRpaBDD
Suite Setup       Given I start deployment "${DEPLOYMENT}"
Suite Teardown    Then I finalize deployment

*** Variables ***
# Named values reused throughout the file
${DEPLOYMENT}    quotes-toscrape
${ENTRY_URL}     https://quotes.toscrape.com/

*** Test Cases ***
# Each test case is a named block of keyword calls
Artifact Catalog
    Given I register artifact "${ARTIFACT_QUOTES}"
    ...    field=quote_text    type=string    required=true

*** Keywords ***
# User-defined keywords (reusable steps) -- optional section

Line-by-line breakdown (from quotes-test.robot)

*** Settings ***
Section header. RF uses this exact format (three stars, name, three stars).
Library Browser
Import the Browser library (Playwright for RF). The spacing is two or more spaces — this is how RF separates columns.
Library WiseRpaBDD
Import our custom Python keyword library. RF will look for WiseRpaBDD.py in the Python path.
Suite Setup Given I start deployment "${DEPLOYMENT}"
Run this keyword once before all test cases in the file. ${DEPLOYMENT} is a variable (see next section).
Watch Out
The separator between columns is two or more spaces (or a tab). A single space is part of the keyword name. This is the #1 source of beginner errors — if your keyword is not recognized, check your spacing.

The ... continuation syntax

When a keyword call has many arguments, you can spread them across lines using ... (three dots) at the start of each continuation line:

    Given I register artifact "${ARTIFACT_QUOTES}"
    ...    field=quote_text    type=string    required=true
    ...    field=author        type=string    required=true
    ...    field=tags          type=array     required=true

All four lines are a single keyword call with multiple arguments.

The optional *** Comments *** section

The project's golden tests start with a *** Comments *** section that documents the evidence from the exploration phase — which selectors exist, how many elements were found, and the pagination structure. RF ignores this section entirely; it is for humans and AI agents.

The ${DEPLOYMENT} variable is then referenced in keywords like:

Suite Setup    Given I start deployment "${DEPLOYMENT}"

RF substitutes ${DEPLOYMENT} with quotes-toscrape before calling the Python function.

Embedded variables in keyword names

WiseRpaBDD uses a special RF feature: embedded arguments. The keyword is defined with "${...}" placeholders right in its name:

@keyword('Given I start deployment "${deployment}"')
def start_deployment(self, deployment: str) -> None:

When RF sees Given I start deployment "quotes-toscrape", it pattern-matches the keyword name, extracts "quotes-toscrape", and passes it as the deployment parameter.

Key Concept
In this project, almost all variables are scalars (${...}). They are used to avoid hard-coding deployment names, URLs, and artifact names — making each .robot file easy to parameterize.

5 Libraries

Importing libraries

In the *** Settings *** section, the Library keyword tells RF to load a keyword library:

Library    Browser       # robotframework-browser (Playwright wrapper)
Library    WiseRpaBDD    # our custom Python library

Three types of libraries

  1. Built-in — BuiltIn, Collections, String, OperatingSystem (shipped with RF).
  2. External — installed via pip: robotframework-browser (Playwright), robotframework-requests, etc.
  3. Custom — Python files you write. RF finds them via PYTHONPATH or the --pythonpath CLI flag.

How RF finds WiseRpaBDD

When you run:

robot --pythonpath scripts/ tests/golden/quotes-test.robot

RF adds scripts/ to the Python path, sees Library WiseRpaBDD, and imports scripts/WiseRpaBDD.py. All methods decorated with @keyword become available as RF keywords.

Run python -c "import WiseRpaBDD; print(dir(WiseRpaBDD.WiseRpaBDD))" from the scripts/ directory to see all methods that become RF keywords.

6 Suite Setup / Teardown

Lifecycle hooks

RF has four hook points:

HookWhen it runsUsage in WiseRpaBDD
Suite SetupOnce, before all test cases in the fileGiven I start deployment "${DEPLOYMENT}"
Suite TeardownOnce, after all test cases finishThen I finalize deployment
Test Setup / [Setup]Before each individual test caseGiven I start resource "..." at "..."
Test Teardown / [Teardown]After each individual test case(Not used in this project)

The WiseRpaBDD lifecycle

# This runs FIRST -- creates the DeploymentContext
Suite Setup       Given I start deployment "${DEPLOYMENT}"

# Test cases run in file order, building the plan:
# - Artifact Catalog  (registers what data we want)
# - Resource cases    (define rules, expansions, extraction)
# - Quality Gates     (set validation thresholds)

# This runs LAST -- executes the entire plan via the browser
Suite Teardown    Then I finalize deployment
Key Concept
The Suite Teardown is where all the real work happens. Every keyword before it just builds an in-memory plan. The finalize_deployment keyword creates an ExecutionEngine and launches the browser. This is the deferred execution model (covered in depth in Topic 16).

Test-level [Setup]

Individual test cases can have their own setup using the [Setup] tag. In WiseRpaBDD, resource test cases use this to initialize a resource context:

Resource quote_pages
    [Setup]    Given I start resource "quote_pages" at "${ENTRY_URL}"

This creates a new ResourceContext object scoped to that test case. All subsequent keywords in that test case (rules, expansions, extraction) attach to this resource.

9 robotframework-browser Basics

robotframework-browser is RF's Playwright wrapper. It provides keywords like New Browser, New Page, Click, and Get Text.

The browser lifecycle

# 1. Launch a browser instance (Chromium, Firefox, or WebKit)
New Browser    chromium    headless=true

# 2. Create a browser context (like an incognito window -- isolated cookies/storage)
New Context

# 3. Open a page (tab) and navigate
New Page    https://quotes.toscrape.com/

# ... do things ...

# 4. Clean up
Close Context
Close Browser

How WiseRpaBDD uses this

You never call New Browser directly in a WiseRpaBDD .robot file. The ExecutionEngine calls these keywords programmatically from Python:

# From ExecutionEngine.run() in WiseRpaBDD.py:
bl = self._bl()                       # Get the browser adapter
bl.new_browser(headless=not self.headed)  # Launch browser
# ... for each resource:
bl.new_context(**ctx_opts)            # New incognito context
bl.new_page()                         # New tab
bl.go_to(entry_url)                   # Navigate
# ... run rules ...
bl.close_context()                    # Clean up context
Key Concept
The .robot file declares what to scrape. The ExecutionEngine handles how to drive the browser. You never write New Browser or Click in a golden test — those are implementation details hidden inside the engine.

The >> chaining operator

Playwright's >> chains selectors so each part scopes into the previous match. The execution engine uses this extensively:

# In _expand_elements, to select the 3rd quote element:
elem_selector = f"{selector} >> nth={i}"
# Result: "div.quote >> nth=2"

# In _extract_field, to find text INSIDE that element:
selector = f"{scope} >> {fs.locator}"
# Result: "div.quote >> nth=2 >> .text"

This is how each field is extracted relative to its parent element, not from the entire page.

11 Waiting Strategies

Why waiting matters

Web pages load asynchronously. An element might not exist yet when you try to click it. Playwright provides several wait strategies:

StrategyRF-Browser KeywordWhen to use
Load stateWait For Load StateAfter navigation — wait for page to finish loading
Element stateWait For Elements StateWait for a specific element to appear/disappear
Auto-wait(built-in to Click, Fill, etc.)Playwright auto-waits for elements before interacting
TimeoutSet Browser TimeoutGlobal timeout for all operations

How the engine uses waits

# After clicking the "Next" pagination button:
bl.click(next_selector)
bl.wait_for_load_state(self._PageLoadStates.domcontentloaded)
# Then an additional delay from resource globals:
page_delay = int(res.globals_.get("page_load_delay_ms", 0))
if page_delay:
    time.sleep(page_delay / 1000.0)

Load states explained

  • domcontentloaded — HTML has been parsed (fastest, used after pagination clicks)
  • load — all resources (images, stylesheets) loaded
  • networkidle — no network activity for 500ms (slowest, used for AJAX-heavy pages)

The page_load_delay_ms resource global

In the .robot file, you set an additional delay:

    And I set resource globals
    ...    timeout_ms=30000           # max wait for any operation
    ...    retries=2                  # retry failed operations twice
    ...    page_load_delay_ms=1000    # extra 1-second wait after navigation

This helps with sites that have JavaScript rendering delays beyond what domcontentloaded captures.

The extraction pipeline

Here is the chain from .robot declaration to actual data extraction:

# In the .robot file you declare:
Then I extract fields
...    field=quote_text    extractor=text       locator=".text"
...    field=author        extractor=text       locator="small.author"
...    field=tags          extractor=grouped    locator=".tag"

# The engine calls _extract_field for each FieldSpec:
if fs.extractor == "text":
    return bl.get_text(selector).strip()    # Get visible text
elif fs.extractor == "attr":
    return bl.get_attribute(selector, fs.attr)  # Get HTML attribute
elif fs.extractor == "grouped":
    # Loop over all matching elements, collect text into a list
    for i in range(count):
        text = bl.get_text(f"{selector} >> nth={i}").strip()
        results.append(text)
    return results

The self-selector: locator="."

The special locator "." means "the element itself" rather than a child. When used with extractor=link it returns the current page URL. When used with extractor=text on a scoped element it returns the element's own text.

Fallback selectors: selector1 | selector2

When a site has inconsistent DOM across pages (e.g., .price-new on sale items, .price on regular items), you can provide multiple selectors separated by | (space-pipe-space). The engine tries each in order and uses the first that matches:

    Then I extract fields
    ...    field=price    extractor=text    locator=".price-new | .price | .amount"

The _resolve_fallback_selector method checks each candidate via get_element_count and returns the first with matches. If none match, it falls back to the first candidate (preserving error messages). Plain selectors (no pipe) pass through with zero overhead.

Passthrough keywords

For cases not covered by the built-in action keywords, two escape hatches allow deferring arbitrary calls into the rule walk:

# Call any method on the browser adapter directly:
    And I browser step "Press Keys"    "#search"    "Enter"

# Invoke any Robot Framework keyword (browser is live):
    And I call keyword "Login To Site"

Both execute during the rule walk when the browser is live. browser step calls a method on the adapter by name. call keyword uses RF's BuiltIn().run_keyword() to invoke any keyword — useful for complex auth flows that need raw Browser library calls.

Watch Out
Passthrough keywords bypass the generic keyword contract. Use them sparingly — they make suites less portable across runtimes and harder for AI agents to reason about.

13 Python Class as RF Library

The @library and @keyword decorators

A decorator is a Python feature that modifies a function or class. It starts with @ and sits on the line above the thing it decorates. Here is the WiseRpaBDD class declaration:

from robot.api.deco import keyword, library

@library(scope="SUITE", auto_keywords=False)
class WiseRpaBDD:
    def __init__(self) -> None:
        self.events: list[tuple[str, tuple[Any, ...]]] = []
        self._deployment: DeploymentContext | None = None
        self._current_resource: ResourceContext | None = None
        self._current_rule: RuleNode | None = None
@library(scope="SUITE", auto_keywords=False)
@library tells RF "this class is a keyword library." scope="SUITE" means one instance is created per .robot file (not per test case). auto_keywords=False means only methods with @keyword are exposed — not every public method.
class WiseRpaBDD:
A Python class. RF instantiates it automatically when Library WiseRpaBDD appears in Settings.
self._deployment: DeploymentContext | None = None
Instance variable with a type hint. DeploymentContext | None means "this can be a DeploymentContext or None." The underscore prefix _ is a Python convention meaning "private — not part of the public API."

Dataclasses

A dataclass is a Python shortcut for creating classes that mainly hold data:

from dataclasses import dataclass, field

@dataclass
class FieldSpec:
    name: str                       # Required field, no default
    extractor: str                  # Required: text, attr, grouped, etc.

15 The from __future__ import annotations Pattern

What it does

The very first non-comment line in WiseRpaBDD.py is:

from __future__ import annotations

This enables postponed evaluation of annotations (PEP 563). Without it, Python evaluates type hints at import time. That means this would fail:

@dataclass
class RuleNode:
    children: list[RuleNode] = field(default_factory=list)
    # ERROR without future annotations: RuleNode is not yet defined!

With from __future__ import annotations, Python treats all type hints as strings (evaluated lazily), so you can reference the class inside its own definition.

The str | None syntax

This also enables the | union syntax for type hints in Python 3.9/3.10+. Without the import, you would need:

from typing import Optional
attr: Optional[str] = None   # Older style

# With future annotations, the cleaner syntax works:
attr: str | None = None

16 The Deferred Execution Model

The fundamental insight

RF executes keywords sequentially, one at a time. But web scraping requires nested loops: for each page, for each element on the page, extract multiple fields. You cannot express nested loops in RF's flat keyword sequence.

The solution is a two-phase architecture:

PhaseWhat happensWhen
Phase A: PlanRF keywords build an in-memory tree of rules, expansions, and field specsDuring test case execution (before Suite Teardown)
Phase B: Executefinalize_deployment walks the tree using a real browserDuring Suite Teardown

How keywords build the plan

# Phase A: Each keyword just stores data on the context

@keyword('Given I start deployment "${deployment}"')
def start_deployment(self, deployment):
    self._deployment = DeploymentContext(name=deployment)
    # Creates the empty plan object

@keyword('Given I register artifact "${artifact}"')
def register_artifact(self, artifact, *fields):

18 The Rule Tree

What is a rule tree?

Each resource has a tree of RuleNodes. Rules have parent-child relationships declared via I declare parents. The engine walks this tree top-down, executing state checks, expansions, and extractions at each node.

Topological sort & diamond DAGs

Before execution, _build_rule_tree populates each node's children list and runs a topological sort (Kahn's algorithm) to determine execution order. This handles:

  • Diamond DAGs — a rule with multiple parents executes exactly once, after all parents have run
  • Cycle detection — raises ValueError if rules form a cycle
  • Linear chains — rules execute in parent-before-child order

Retry mechanism

Rules can declare retry behavior for transient failures:

    And I set retry 3 times with 500 ms delay

This sets rule.retry_max = 3 and rule.retry_delay_ms = 500. The engine will retry the entire rule walk up to 3 times if extraction fails.

19 The Browser Adapter Pattern

The problem

Some websites (Yelp, Cloudflare-protected sites) detect automated browsers and block them. The robotframework-browser library does not support Playwright's context.addInitScript(), which is needed to inject anti-detection patches before page JavaScript runs.

The solution: two backends

BackendClassWhen used
RF-Browser (default for non-stealth)Direct library instance from _get_rf_browser()WISE_RPA_STEALTH=false
Stealth adapter_StealthAdapterWISE_RPA_STEALTH=true (default)

How _bl() picks the backend

def _bl(self) -> Any:
    """Lazy-init the browser backend."""
    if self._adapter is None:
        if self.stealth:
            self._adapter = _StealthAdapter()
        else:
            self._adapter = _get_rf_browser()
    return self._adapter

Both backends expose the same method names (click, get_text, get_attribute, get_element_count, go_to, new_browser, new_context, etc.), so the engine does not need to know which one it is talking to.

_StealthAdapter highlights

class _StealthAdapter:
    def new_browser(self, *, headless=True):
        try:
            from patchright.sync_api import sync_playwright
        except ImportError:
            from playwright.sync_api import sync_playwright
        self._pw = sync_playwright().start()
        self._browser = self._pw.chromium.launch(
            headless=headless, channel="msedge",
            args=["--disable-blink-features=AutomationControlled"])

    def new_context(self, **kw):
        from playwright_stealth import Stealth
        self._context = self._browser.new_context(**merged)
        Stealth().apply_stealth_sync(self._context)
        # ^ THIS is what RF-Browser cannot do
Key Concept
This is the Adapter pattern from software design. Both backends "adapt" to the same interface. The WISE_RPA_STEALTH environment variable controls which one is used. Default is stealth ON.

Patchright: next-generation stealth

The stealth adapter now tries to import patchright before falling back to playwright. Patchright is a Playwright fork that patches Chromium at the binary level for undetectable automation — it defeats TLS/JA3 fingerprinting and browser-level automation flags that playwright-stealth alone cannot address. The adapter logs which backend is active:

19b Browser Control Hierarchy

The engine has a 3-layer stack for browser control. Understanding which layer owns the page in each mode is essential for debugging and for writing call_keyword blocks.

         ↓ dispatches to
Layer 2: Adapter Interface    click(), fill_text(), get_text(), …
         ↓ implemented by
Layer 1a: RFBrowserAdapter    → RF-Browser lib → Playwright (via gRPC to Node.js)
Layer 1b: StealthAdapter      → Patchright (patched Playwright, in-process Python)

Which layer owns the page?

ModePage ownerRF keywords work?
Non-stealth (default)Layer 1a — RF-Browser's Node.js processYes, via gRPC
StealthLayer 1b — Patchright's in-process Python pageOnly via stealth bridge

The call_keyword problem

And I call keyword "Login" runs an RF keyword at walk time. Inside that keyword, raw RF-Browser calls (Click, Fill Text) go directly to Layer 3. In stealth mode, Layer 3 has no page — the page lives in Layer 1b.

The stealth bridge solution

The _StealthBrowserBridge class implements RF's dynamic library interface and delegates to the stealth adapter. When a call_keyword runs in stealth mode, the engine temporarily swaps the Browser library's internal instance with the bridge. RF's keyword resolution re-evaluates owner.instance on each keyword call (not cached), so all Browser keywords route to the stealth adapter's live page for the duration of the call.

Key Concept
RF-Browser stores pages in a Node.js gRPC server, not in Python. You cannot inject a patchright page into RF-Browser's catalog. The bridge approach (instance swap) is the only viable strategy at the Python level.

20 Expansion Strategies

Expansions are how the engine loops. The .robot file declares what to loop over, and the ExecutionEngine handles the how.

1. Elements expansion (elements)

Loop over N matching CSS elements on the current page. The most common expansion type.

# .robot declaration:

# Creates: Expansion(over="elements", scope=".quote")

Engine behavior (_expand_elements):

count = bl.get_element_count(selector)   # How many .quote on page?
for i in range(count):
    elem_selector = f"{selector} >> nth={i}"    # ".quote >> nth=0", etc.
    record = self._extract_from_element(rule, elem_selector, ...)
    # Then walk children with this element as scope

BFS vs DFS order:

# DFS (default): process each element fully before moving to next
When I expand over elements ".quote"

# BFS: extract ALL elements first, then walk children
# Used when children navigate away (e.g., clicking product links)
When I expand over elements ".thumbnail" with order "bfs"
...    limit=6

BFS extracts all 6 product URLs before navigating to any of them. DFS would click the first product, lose the listing page, and fail to extract the remaining 5.

2. Next-button pagination (pages_next)

Click a "Next" button repeatedly until it disappears or the limit is reached.

# .robot declaration:
When I paginate by next button "li.next a" up to 3 pages

Engine behavior:

for page_num in range(limit):
    for child in rule.children:
        child_data = self._walk_rule(child, ...)
    count = bl.get_element_count(next_selector)
    if count == 0:
        break       # No next button -- pagination ends naturally
    bl.click(next_selector)
    bl.wait_for_load_state(domcontentloaded)

3. Numeric pagination (pages_numeric)

Click numbered page links (1, 2, 3...) instead of a "Next" button.

When I paginate by numeric control "a.page-link" from 1 up to 10 pages

The engine finds the link whose text matches the target page number and clicks it.

4. Combinations expansion (combinations)

Cartesian product of UI control values. Used for variant extraction (e.g., clicking different HDD size buttons).

# From variants-test.robot:
When I expand over combinations
...    action=click    control="button.swatch"    values=128|256|512|1024

The engine computes the full Cartesian product using itertools.product. With one axis of [128, 256, 512, 1024] it produces 4 iterations. With multiple axes (e.g., color AND size) it produces every combination.

Key Concept
For each combination, the engine finds the matching UI control by text comparison, clicks it, waits for any AJAX update, then extracts fields from the resulting page state.

State setup (authentication)

Many sites require login before scraping. The configure state setup keyword defines a sequence of pre-scrape actions:

    Given I configure state setup
    ...    skip_when=.logged-in-indicator
    ...    action=open     url=https://example.com/login
    ...    action=input    css=#username    value=myuser
    ...    action=password css=#password    value=mypass
    ...    action=click    css=button[type=submit]

ActionWhat it does

passwordSame as input but semantically distinct for passwords
clickClick an element (bl.click(css))

The skip_when option checks for a CSS selector — if found, setup is skipped (already logged in). After all actions, the engine waits for networkidle.

Interrupt handling

Cookie banners, popup modals, and consent dialogs can block scraping. The configure interrupts keyword registers selectors to auto-dismiss:

    And I configure interrupts
    ...    dismiss=button#onetrust-accept-btn-handler
    ...    dismiss=.modal-close-button

The engine calls _dismiss_interrupts which checks each selector and clicks it if found. This runs before every action during step execution.

Per-rule dismiss scoping

Global dismiss selectors can interfere with interactive panels (calendars, guest pickers, dropdown menus). Two keywords let you scope or suppress dismiss per rule:

    # Override: use ONLY these selectors for this rule
    I define rule "interactive_form"
        And I scope interrupts
        ...    dismiss="button.cookie-accept"
        ...    dismiss=".promo-close"

    # Suppress: skip ALL dismiss for this rule
    I define rule "datepicker_interaction"
        And I pause interrupts
        When I click locator ".calendar-next"

When a rule declares And I scope interrupts, only those selectors are used for dismiss during that rule's steps. When And I pause interrupts is declared, no dismiss runs at all. After the rule completes, the engine returns to the global config.

Declarative rule options

Rules can declare lifecycle hooks via And I set rule options:

    I define rule "checkout"
        And I set rule options
        ...    on_enter=screenshot
        ...    on_fail=screenshot
        ...    timeout_ms=30000
OptionEffect
on_enter=screenshotTake screenshot before guard checks
on_fail=screenshotTake screenshot on guard failure or rule timeout
timeout_ms=NAbort rule after N milliseconds
Watch Out
Interrupt selectors use simple get_element_count + click. If a dismiss button triggers a page reload, your extraction timing may be affected. Use page_load_delay_ms as a safety buffer. Use And I pause interrupts on rules where dismiss would break the flow.

22b Observation Gates & Zero-Sleep Execution

WiseRpaBDD models every scrape as a delayed MDP: state → action → observation. Fixed-duration waits (When I wait) are an anti-pattern because they paper over missing observations. This section covers the two observation-gate patterns that replace all sleeps, the scoping rules for dismiss selectors, and how the engine achieves zero-sleep execution internally.

1. The Problem: Why When I wait Is an Anti-Pattern

A When I wait 3 seconds step tells the engine nothing about what it is waiting for. It introduces three problems:

  • Hidden observations. The MDP transition s, a → o is incomplete — the observation o is implicit ("something probably changed after 3 seconds"). The engine cannot verify the transition succeeded.
  • Fragility. Too short and the test is flaky on slow networks; too long and the suite is needlessly slow. There is no correct fixed value.
  • Engine violation. The engine should never call time.sleep. All timing is event-driven: Playwright's wait_for_elements_state (default 10s timeout), staleness detection, or explicit selector polling.
Key Concept
Every action must declare its observation gate — the selector or condition that proves the action's effect is visible in the DOM. If you cannot name the gate, the action's MDP node is incomplete.

2. Option 1: Split Rules with State Gates

The purest MDP representation: each s, a → o transition is its own rule node. The engine's built-in wait_for_elements_state (10s default) handles timing at each node boundary.

Example: An Airbnb-style city autocomplete flow. Typing a city name triggers an async autocomplete dropdown. The three-rule split makes every observation explicit:

I define rule "type_city"
    And I declare parents "expand_search"
    When I type "${CITY_SEARCH}" into locator "#bigsearch-query-location-input"

I define rule "autocomplete_ready"
    And I declare parents "type_city"
    And selector "[data-testid='option-0']" exists

I define rule "select_city"
    And I declare parents "autocomplete_ready"
    When I click locator "[data-testid='option-0']"

When to use: Named state transitions worth documenting — login flows, multi-step wizards, any place where the intermediate state is meaningful to a reader of the .robot file.

Try It
Check if an existing When I wait can be replaced by splitting the rule. If the wait is between "type" and "click", there is almost certainly a selector you can gate on.

3. Option 2: await= Inline Observation Gate

When splitting into separate rules adds noise without clarity, use the await= continuation parameter. It keeps rules grouped by user intent while still declaring every observation.

await=<selector> can be attached to any action keyword (click, type, click_text). The engine waits for the selector to appear in the DOM before advancing to the next action. Fallback selectors are supported via the standard | separator.

Example: The same Airbnb city flow, compact form:

I define rule "enter_city"
    When I click text "Anywhere"
    ...    await=#bigsearch-query-location-input
    When I type "${CITY_SEARCH}" into locator "#bigsearch-query-location-input"
    ...    await=[data-testid='option-0']
    When I click locator "[data-testid='option-0']"

When to use: Sequences where the intermediate states are mechanical (click → wait for input to appear) and naming each one would clutter the rule tree.

Key Concept
Both options produce zero-sleep execution. Option 1 (split rules) is self-documenting but verbose. Option 2 (await=) is compact but hides observations inside continuation lines. Choose based on whether readers need to see each state.

4. Dismiss Selector Scoping

The dismiss mechanism (I configure interrupts ... dismiss=<selector>) runs before every action in the rule walk. This is powerful but dangerous: an overly broad selector will close interactive panels the user intentionally opened.

Watch Out
CRITICAL: [role="dialog"] matches search panels, date-picker calendars, and filter drawers — not just cookie banners. A dismiss selector that clicks the close button of any [role="dialog"] will sabotage your own flow.

Rules for dismiss selectors:

  • Only dismiss known popup text or patterns — be as specific as possible.
  • Test your dismiss selectors during the explore phase with all interactive panels open.
  • Prefer text-based selectors that match the popup's accept/dismiss button label.
QualitySelectorWhy
Goodtext="Got it"Matches only the specific cookie-consent button text
Goodbutton#onetrust-accept-btn-handlerID-specific, cannot match anything else
Bad[role="dialog"] button[aria-label="Close"]Matches every dialog's close button, including calendars and search panels
Bad.modal-overlay .closeToo generic — any component using modal patterns will be dismissed

5. Engine Internals: How Zero-Sleep Works

The engine achieves zero-sleep execution through four mechanisms. None use time.sleep.

MechanismWhere UsedHow It Works
Staleness detectionPaginationAfter clicking "next page", the engine polls until the first element's text content changes from its pre-click value. This proves the new page has rendered.
Heading-text pollingCalendar forwardAfter clicking the calendar's forward button, the engine polls until the month/year heading text differs from its pre-click value.
JS click + wait_for_elements_stateStepper / quantity controlsStandard Playwright click() stability checks fail on elements that re-render during the click. The engine uses a raw JS click followed by wait_for_elements_state("visible") as a post-condition.
AOP instrumentationAll actionsThe _do_action_instrumented wrapper runs around every action. It handles pre-action dismiss, post-action observation gates (await=), and retry/recovery. Enabled by default.
Key Concept
The AOP wrapper (_do_action_instrumented) is the single point of control for all action timing. It runs dismiss selectors, executes the action, waits for the await= gate if present, and retries on failure. Every action in the rule walk passes through this wrapper — there is no path that bypasses it.

24 Emit Patterns

After extraction, records must be emitted to an artifact store. WiseRpaBDD supports three emit strategies controlled by the keyword used.

Simple emit

    And I emit to artifact "${ARTIFACT_QUOTES}"

Appends each extracted record directly to the named artifact's store.

Flattened emit

    And I emit to artifact "${ARTIFACT_QUOTES_FLAT}" flattened by "tags"

If a field contains a list (e.g., tags: ["love", "life", "humor"]), the engine creates one record per list item. A quote with 3 tags becomes 3 flat records. Useful for generating denormalized/tabular output.

Merge emit

    And I merge into artifact "${ARTIFACT_PRODUCTS}" on key "product_id"

If a record with the same key value already exists in the artifact store, the new data is merged into the existing record (via dict.update). If no match is found, the record is appended normally. This is used when multiple rules or expansions contribute fields to the same logical entity.

Direct write

    Then I write artifact "${ARTIFACT}" to "output/my_data.json"

Writes an artifact's current contents to a specific file path, useful for mid-suite checkpoints or non-standard output locations.

Artifact options affecting output

    And I set artifact options for "${ARTIFACT}"
    ...    output=true              # write to disk (default: false)
    ...    structure=flat            # flat vs nested (default: nested)
    ...    format=jsonl              # json, jsonl, csv, markdown
    ...    dedupe=product_id         # deduplicate on this field
    ...    query=[].data             # JMESPath transform before writing
    ...    consumes=product_urls     # declares dependency on another artifact

25 Multi-Resource Chaining

A deployment can have multiple resources that form a pipeline. Resources can consume artifacts produced by earlier resources, enabling multi-step scraping.

Example: variants-test.robot

# Resource 1: "discover" -- scrapes product URLs
Resource discover
    [Setup]    Given I start resource "discover" at "${ENTRY_DISCOVER}"
    # ... extracts product URLs, emits to "product_urls" artifact

# Resource 2: "variants" -- consumes product URLs, visits each one
Resource variants
    [Setup]    Given I start resource "variants" at "${ENTRY_VARIANTS}"
    # ${ENTRY_VARIANTS} = "https://www.webscraper.io{url}"
    # The {url} is a template that gets expanded per product_urls record

How the engine resolves execution order

The engine uses Kahn's algorithm (topological sort) on the artifact dependency graph:

  1. Build a directed graph: resource B depends on resource A if B consumes an artifact that A produces
  2. Run topological sort to produce an execution order (e.g., ["discover", "variants"])
  3. Execute resources in that order

Template URL expansion

When a resource entry URL contains {field}, the engine expands it for each record in the consumed artifact:

# Template: "https://www.webscraper.io{url}"
# Consumed artifact records: [{"data": {"url": "/product/60"}}, ...]
# Expanded: "https://www.webscraper.io/product/60"
# The engine visits each expanded URL in sequence

Alternative chaining: iterate over parent records

    Given I iterate over parent records from "discover"

Instead of URL templates, this approach passes each parent record directly to the child resource, which can access parent data during extraction.

The complete execution flow

ExecutionEngine.run()
    |-- new_browser()
    |-- _execute_resources()              # Topologically sorted
    |     |-- _execute_resource(discover)
    |     |     |-- new_context(), new_page()
    |     |     |-- go_to(entry_url)
    |     |     |-- _build_rule_tree()    # Populate children, topo sort
    |     |     |-- _walk_rule(root)
    |     |     |     |-- _check_state()
    |     |     |     |-- _handle_expansion(elements, bfs)
    |     |     |     |     |-- _extract_from_element() x N
    |     |     |     |-- _emit_records("product_urls")
    |     |     |-- close_context()
    |     |
    |     |-- _execute_resource(variants)
    |           |-- _resolve_entry_urls() # Template expansion from product_urls
    |           |-- For each URL:
    |                 |-- go_to(url)
    |                 |-- _walk_rule(detail_root)
    |                       |-- _handle_expansion(combinations)
    |                             |-- Click swatch, extract, repeat

Quality gates in .robot

How quality gates are checked

After all resources are scraped and output is written, the engine calls _check_quality_gates:

  1. Min records — total records across all artifacts must meet the threshold
  2. Field fill % — for each declared field, the percentage of records with a non-empty value must meet the threshold
  3. Max failed % — the percentage of failed extractions (empty records) must be below the threshold
Watch Out
