Case 1: Simple Extraction (no artifacts)

webscraper.io laptops — 117 products on one page. Single resource, no chaining needed.

laptop-products-scrape
https://www.webscraper.io/test-sites/e-commerce/allinone/computers/laptops
root state check
state: url_pattern, .thumbnail exists
products expand × 117
expand: elements, scope=".thumbnail"
extract: title, price, desc, rating

Runtime trace

1. open URL → interrupt.check() → cookie dismissed
2. [node] root → state check passes
3. [node] products → expand elements → 117 .thumbnail matches
4. for each match: extract {title, price, description, rating} → push record
5. → 118 records → JSONL

Data flow: Node → records[] → JSONL file. No artifacts involved.

Case 2: Paginated Extraction (expand over pages)

webscraper.io laptops (static) — 20 pages × 6 products. Three-node DAG with page expansion.

laptop-paginated-scrape
https://www.webscraper.io/test-sites/e-commerce/static/computers/laptops
root
state: .thumbnail exists
pages expand × 20
expand: pages, strategy=next
control: a.page-link.next
products expand × 6/page
expand: elements, scope=".thumbnail"
extract: title, price, desc, rating

Runtime trace (DFS — default)

1. [node] root → state passes
2. [node] pages → expand pages (DFS)
│ Page 1: walk children → [node] products → extract 6 rows
│ Page 2: click next → walk children → extract 6 rows
│ ...
│ Page 20: next button gone → stop
3. → 117 records → JSONL

Data flow: Nested expansion (pages × elements). Each page is a successor state; within each, elements expand. Still no artifacts — everything stays in the records[] array.

Case 3: Multi-Phase Pipeline (artifacts link resources)

Splunk ITSI docs — discover 80+ pages from nav, then extract each. Two resources linked by artifact.

splunk-itsi-administer
https://help.splunk.com/.../about-administering-it-service-intelligence
internal page_urls → { url: string, title: string }
output page_content → { title: string, body: string } → markdown
Resource: discover
produces: page_urls
node: toc
expand: elements, order: bfs
extract: url, title
Resource: extract_pages
consumes: page_urls
produces: page_content
entry.url: { from: "page_urls" }
extract: title, body

Runtime linking trace

0. resolveOrder() → topological sort → [discover, extract_pages]
produces/consumes graph: discover→page_urls→extract_pages

1. Resource: discover
│ open entry URL
│ [node] toc → expand 80 elements (BFS)
│ for each nav link: extract {url, title}
│ → 80 records returned
│ store.put("page_urls", 80 records) ← resource.produces writes to store

2. Resource: extract_pages
│ resolveEntryUrls() → entry.url is {from: "page_urls"}
│ store.getUrls("page_urls") ← reads url field from each record
│ → 80 URLs
│ for i = 0..79:
│ │ open URL[i]
│ │ [node] page → extract {title, body}
│ → 80 records returned
│ store.put("page_content", 80 records)

3. Output
│ page_content has output: true, format: markdown
│ → write splunk-itsi_page_content.md

Case 4: Node-Level Chaining (single resource, yields/consumes)

Same Splunk use case, but using node-level data flow instead of two resources.

Single-resource variant (equivalent)
internal page_urls → toc yields → pages consumes
root
toc yields
expand: elements, order: bfs
extract: url, title
yields: page_urls
pages consumes
consumes: page_urls
action: navigate {url}
extract: title, body

Runtime linking trace

1. [node] root → walk children in YAML order

2. [node] toc (first child of root)
│ expand 80 elements (BFS) → extract {url, title} × 80
│ store.put("page_urls", [record]) ← called 80 times via node.yields
│ BFS: all 80 records in artifact BEFORE children walk

3. [node] pages (second child of root — runs after toc)
│ store.get("page_urls") → 80 records
│ for each consumed record:
│ │ consumedData = {url: "https://...", title: "..."}
│ │ navigate → resolveUrl("{url}") → checks consumedData first
│ │ extract {title, body}
│ → 80 records

⚠ Ordering depends on YAML declaration order. If "pages" were listed before "toc" in the YAML, it would run first and see an empty artifact. No topological sort for node-level chaining — the exploration agent must emit nodes in the correct order.

How Linking Works at Each Level

Cross-Resource (materialized)

At startup, ArtifactStore.resolveOrder() builds a DAG from all resources' produces/consumes fields. Kahn's algorithm gives execution order. Resources run sequentially in that order.

Linking: store.put(name, records) after producer completes. store.getUrls(name) when consumer resolves entry URLs. The store is a Map<string, Record[]>.

✓ Ordering guaranteed by topological sort

Node-Level (logical, YAML-ordered)

No dependency resolution. Engine walks children of each parent in YAML array order. A yielding node writes to store.put() during expansion. A consuming sibling reads via store.get().

Linking: Same ArtifactStore instance shared across all nodes. yields writes per-record. consumes reads the full artifact and iterates.

⚠ Ordering depends on YAML declaration order

The ArtifactStore at Runtime

// Shared across entire profile execution
ArtifactStore {
  store: Map {
    "page_urls" → [{url, title}, {url, title}, ... × 80]
    "page_content" → [{title, body}, {title, body}, ... × 80]
  }
  schemas: {
    "page_urls": { fields: {url: required, title: required} }
    "page_content": { fields: {title: required, body: required}, output: true }
  }
}

// Write path (producer):
store.put("page_urls", records)
  → validate each record against schema.fields
  → append to store.get("page_urls")
  → log validation errors if any

// Read path (consumer):
store.getUrls("page_urls")
  → map records: prefer data.url, fallback record.url
  → return string[]

store.get("page_urls")
  → return ExtractedRecord[] (full records for iteration)

Decision Guide: When to Use What

Pattern Mechanism When
Extract from single page expand: elements All data on one page
Paginate + extract expand: pages → child expand: elements Data split across pages
Discover URLs → extract each Node yields + sibling consumes (intra-resource) Same globals, simple pipeline
Discover → extract (separate config) Resource produces/consumes (cross-resource) Different timeouts, setup, hooks per phase
Multi-section doc site Multiple resources, each produces same artifact 3 sections → merge into one output
